Operon

"An AI-powered workflow orchestration system that converts natural language instructions into safe, sandboxed, auditable system actions — using a locally hosted LLM and an MCP security layer. No cloud. No API keys."

Problem Statement

Existing AI assistants like ChatGPT or Claude only generate text. When a user says "set up a Python project," they still have to execute every step manually. There is no controlled, auditable way for an AI to safely carry out real system actions without human intervention — and without risking security breaches.

What Exists Today

ChatGPT, Gemini, Claude — text responses only. Zapier, n8n — automation but needs manual setup. No tool combines natural language AI with real, controlled execution in a secure sandbox.

Our Innovation

Operon separates AI decision-making from system execution. The LLM plans; the MCP Server validates; the Docker Sandbox executes — safely. Every step is logged. Nothing runs without a security check.

Frontend Layer

React SPA served by nginx. Renders live task graph, audit log, run history, MCP inspector, and threat simulator. Connects via Server-Sent Events (SSE) for real-time updates.

Orchestration Layer

Node.js + Express. Sends user instruction to Ollama, receives a JSON task plan, builds the DAG, applies topological sort, and submits tasks one-by-one to the MCP Server.

Security + Execution Layer

Python FastAPI (MCP Server) validates every action against a security policy, then executes approved commands in an isolated Docker container. All decisions logged to SQLite.

What is a Task DAG?

When you type "Create a Python project called my-app", the LLM does not run a single command. It breaks that instruction into a Directed Acyclic Graph (DAG) — a list of tasks where each task may depend on a previous one.

The Orchestrator resolves the order automatically. Tasks with no dependencies run first. Dependent tasks wait. This ensures nothing executes out of sequence.

1
createFolder(my-app)
No dependencies — runs first
2
createFile(my-app/README.md)
Depends on task 1
3
createFile(my-app/requirements.txt)
Depends on task 1
4
initGit(my-app)
Depends on tasks 2 and 3

Key Features

Live Task Graph

The DAG visualizes in real-time on the frontend. Each node updates its color (pending → running → completed / failed) as tasks execute, streamed via SSE.

Audit Log with AI Reasoning

Every MCP security decision — allowed or blocked — is logged with an AI-generated explanation of why the action was permitted or denied.

AI Explainer

After each run, Ollama generates a plain-English summary describing what the workflow accomplished — accessible to non-technical users.

Run History

All past runs are persisted in SQLite and browsable from the UI — with original instruction, status, timestamps, and the AI summary.

MCP Inspector

A built-in panel that lists every registered MCP tool with its full JSON schema — so users and developers can see exactly what actions the system can perform.

Threat Simulator

A live demo that fires 4 real attack payloads at the MCP Server and shows them being blocked in real-time with reasoning — path traversal, system file access, prompt injection, privilege escalation.

MCP Server Security Layer

How Every Action is Validated

Step 1 — Tool Lookup

Is the requested tool in the registered tool catalogue? Unknown tools are immediately rejected.

Step 2 — Schema Validation

Parameters are checked against the tool's JSON schema — correct types, required fields, no missing data.

Step 3 — Security Policy Check

Path traversal patterns (../) are detected. Absolute paths outside /workspace are blocked. Dangerous shell commands are blocked.

Step 4 — Audit + Execute

Decision (allow/block) + AI reasoning written to SQLite. If allowed, command runs in Docker Sandbox and result is returned.

4 Attack Vectors Blocked

Path Traversal

Attempt to read ../../etc/passwd — blocked, ../ pattern detected.

System Directory Access

Attempt to access /etc/shadow or other protected system paths — blocked.

Prompt Injection

Instruction designed to override the LLM's system behaviour — detected and blocked.

Privilege Escalation

Action requesting elevated permissions beyond the sandbox boundary — blocked.

Challenges Faced

•Getting the LLM to consistently return well-structured JSON task plans (not free-form text)
•Designing a security policy that blocks attacks without blocking legitimate tasks
•Handling Docker service startup order — health checks were essential
•Streaming real-time updates from Node.js to React without losing events
•Making the system work offline with a locally hosted model (no cloud APIs)

Key Learnings

•Separating AI decision-making from execution is critical for security
•Docker Compose health checks are not optional in multi-container systems
•Local LLMs (Ollama) can replace cloud APIs effectively for structured tasks
•An audit trail is as important as the feature itself — especially for AI systems
•MCP provides excellent control and transparency over what an AI can actually do

Future Enhancements

Project Outcome

Operon proves that a fully offline, privacy-preserving AI workflow system is achievable with open-source tools alone. It successfully converts natural language into sandboxed, validated, auditable system actions — blocking all 4 tested attack vectors — without needing any cloud API or external credentials.
